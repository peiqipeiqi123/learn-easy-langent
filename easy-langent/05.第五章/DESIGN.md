# 学习助手 Agent — 开发设计文档

> 本文档记录从零到一设计和实现"学习助手 Agent"的完整思路，包括产品定义、Tool 选型、记忆架构、技术决策及迭代过程。

---

## 一、产品定义

### 1.1 用户与场景

**目标用户**：研究者 / 学习者（首先是自己）

**核心痛点**：
- PDF/论文分散在各处，查阅分析效率低
- 缺乏结构化对比和写作辅助
- 对话中灵感突现时无法立即加载新资料
- 知识存在滞后，需要联网补充

**一句话定义**：对话中动态加载文档、随时提问分析、得到带来源引用的可信回答的智能学习助手。

### 1.2 产品定位

"做一个能用的产品，而非一个 Demo"。设计初期就明确了两条原则：

| 原则 | 含义 |
|------|------|
| **练习优先** | 先跑通核心流程，工程化留到后续迭代 |
| **产品思维** | 功能选择从真实使用场景出发，不做无意义堆砌 |

## 二、功能选型

### 2.1 Tool 清单（10 个）

经过多轮讨论，最终确定了 10 个 Tool，分为四类：

**文档管理（3 个）**

| Tool | 设计理由 |
|------|---------|
| `add_document` | 核心入口。支持 PDF/TXT/MD/DOCX，支持文件夹批量。用魔数检测文件类型而非仅依赖扩展名，避免改名后无法加载。 |
| `list_documents` | 用户需要知道当前知识库里有什么，否则会反复加载同一篇。 |
| `remove_document` | 知识库会膨胀，需要清理机制。 |

**核心分析（3 个）**

| Tool | 设计理由 |
|------|---------|
| `search_documents` | RAG 检索 + LLM 生成，是使用频率最高的 Tool。保留 FAISS L2 距离作为相关度参考，但标注为启发式估算。 |
| `summarize_document` | 单篇结构化摘要。输出 JSON（问题/方法/结论/关键词/贡献），可被程序解析复用。 |
| `compare_documents` | 两篇文档对比。**聊天式对比**是 LLM 的强项——传统工具做不到"这两篇论文的方法有什么异同"这种自然语言对比。附带 Mermaid 关系图。 |

**写作辅助（1 个）**

| Tool | 设计理由 |
|------|---------|
| `write_related_work` | 学术界刚需。基于已加载文献自动分组、定位研究空白。设计为"产品级"——输出可直接放入论文草稿。 |

**信息获取（2 个）**

| Tool | 设计理由 |
|------|---------|
| `web_search` | 对抗知识滞后。最初用 DuckDuckGo HTML 抓取，后因信息不完整（搜不到作者等细节）升级为 `duckduckgo-search` 库。 |
| `fetch_url` | 用户直接给链接时使用。区别于搜索——搜索是"找东西"，抓取是"打开看看"。来自 Ch3 Web Query Bot 的复用。 |

**可视化（1 个）**

| Tool | 设计理由 |
|------|---------|
| `generate_flowchart` | 终端渲染不了图，但 Mermaid 是纯文本——LLM 能生成，用户贴到 VS Code / mermaid.live 即可渲染。比 OCR 或生成图片方案的性价比高得多。 |

### 2.2 未实现但讨论过的功能

| 功能 | 为何不做 |
|------|---------|
| Excel 支持 | 学习助手场景用不上，属于"无意义堆砌" |
| OCR 图片识别 | 方案 B 讨论过，需装系统级 Tesseract。最终选择文档标注 + 留升级路径（多模态模型） |
| Web 图形界面 | 课程定位是 CLI。Mermaid 输出已解决可视化问题 |

## 三、记忆架构

### 3.1 选择分层摘要记忆

三种记忆方案对比：

| 方案 | 优点 | 缺点 | 是否采纳 |
|------|------|------|---------|
| 全量记忆 | 无信息丢失 | Token 爆炸 | ❌ |
| 窗口记忆 | 简单 | 旧信息彻底丢失 | ❌ |
| 摘要记忆 | 压缩保留 | 压缩失真 | ✅ 基础方案 |
| **分层摘要记忆** | 全局脉络 + 文档细节双保留 | 实现稍复杂 | ✅ **最终选择** |

**设计原理**：全局摘要记录"你在研究什么、加载了哪些文档、讨论了什么"，文档摘要记录"每篇论文的独立结构化信息"。两层互不干扰——聊了 50 轮也不会冲掉第一篇论文的方法细节。

### 3.2 记忆交互

用户可通过 `:memory` 查看当前记忆状态，通过 `:reset` 清空，通过 `:session` 切换到不同会话的记忆上下文。记忆对用户透明——不是黑盒。

### 3.3 摘要长度迭代

- 初版：150 字 → 太短，信息丢失
- 终版：300 字 → 平衡信息密度和 Token 消耗

## 四、输出设计

### 4.1 混合输出模式

不统一用一种格式，而是根据任务类型选择：

| 任务类型 | 输出格式 | 理由 |
|---------|---------|------|
| 一般问答 | Markdown | 可读性，用户不需要"读代码" |
| 分析/对比/摘要 | JSON + `sources` | 可回溯来源，可被程序解析 |
| 流程图 | Mermaid 代码块 | 可用 VS Code / mermaid.live 渲染 |

### 4.2 流式输出

从 `invoke()` 改为 `stream()`，边生成边显示。关键设计：Tool 调用检测与流式输出共存——chunk 合并后判断是否有 `tool_calls`。

## 五、会话管理

### 5.1 设计演进

| 版本 | 方案 | 问题 |
|------|------|------|
| v1 | 每次启动自动恢复 | 启动慢（需加载 Qwen 模型重建 FAISS） |
| v2 | 启动询问"是否恢复" | 每次弹窗打断体验 |
| v3 | 启动秒进 + `:restore` 命令 | 手动恢复不够直观 |
| **v4** | **启动秒进 + 自动显示会话列表** | ✅ 像聊天软件——打开是新窗口，旧的在侧边栏 |

### 5.2 自动命名

首轮实质对话后 LLM 自动生成 ≤10 字的会话名（如"研究问题选择之道"），替代无意义的 "default"。用户无需操心命名，会话列表一目了然。

### 5.3 持久化

每轮对话自动保存：`session.json`（记忆状态）+ `chunks.json`（文档分块）。FAISS 索引存到 `faiss_sessions/<hash>/`，绕过 Windows 上 FAISS C++ 层不支持 Unicode 路径的限制。

## 六、技术决策

### 6.1 模型与基础设施

| 组件 | 选择 | 理由 |
|------|------|------|
| LLM | DeepSeek Chat | 免费配额，OpenAI 兼容 API |
| Embedding | Qwen3-Embedding-0.6B（本地 CPU） | 复用 Ch4 已部署模型 |
| 向量库 | FAISS（CPU 版） | 教程标准组件，足够小规模用 |
| 包管理 | uv | 比 pip 快，锁定版本 |

### 6.2 架构选择

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 链编排 | 消息列表 + 手动 Agent 循环 | 比 LCEL 链更灵活，支持流式输出和多轮 Tool 调用 |
| Tool 执行 | `bind_tools` + while 循环（最多 8 轮） | 支持"列举文档→生成流程图"等多步调用 |
| 模板安全 | `_safe_invoke()` 自动转义 `{}` | 文档内容含 JSON 代码块时防止模板注入崩溃 |

### 6.3 踩过的坑

1. **FAISS + Unicode 路径**：`05.第五章` 含中文，FAISS C++ 层 `fopen` 用 ANSI 编码无法打开。解决：FAISS 放项目根目录 `faiss_sessions/`，会话名用 MD5 hash 做目录名。

2. **`{}` 模板注入**：ChatPromptTemplate 把文档内容中的 `{` `}` 当变量占位符。最初只在 prompt 文字中手动转义，后来统一封装 `_safe_invoke()` 函数。

3. **`started_output` 重复声明**：Agent 循环内层多写了一次 `started_output = False`，导致流式标记永远不更新。属于典型的复制粘贴 bug。

4. **`:rename` 丢数据**：`shutil.move` 因 FAISS 文件锁失败，改用 `copytree` + `rmtree`（copy-delete 模式）并先关闭 `_vector_db`。

## 七、项目结构

```
05.第五章/
  ├── README.md              ← 用户文档
  ├── DESIGN.md              ← 你在这里（开发设计文档）
  ├── config.py              ← 环境变量、LLM/Embedding 初始化、路径
  ├── memory.py              ← 分层摘要记忆 + 自动命名
  ├── prompts.py             ← 6 个提示模板
  ├── tools.py               ← 10 个 Tool 定义 + 辅助函数
  ├── chains.py              ← LCEL 参考实现（主程序未使用）
  └── learning_assistant.py  ← 主入口：CLI 循环 + 流式输出 + 会话管理
```

### 8.2 关键数据流

```
用户输入
  → handle_command()：命令直接处理（:sessions, :memory, :reset…）
  → Agent 循环：
      消息列表 + 分层记忆上下文
        → LLM.bind_tools(10 tools).stream()
          → 有 tool_calls → 执行 → 结果回传 → 继续循环
          → 无 tool_calls → 流式输出 → 结束
  → memory.add_turn() → 每 3 轮 update_global_summary()
  → save_to_file() + save_chunks() → 持久化到 .sessions/<name>/
```

### 8.3 状态存储

| 数据 | 位置 | 格式 |
|------|------|------|
| 记忆 + 文档注册 | `.sessions/<name>/session.json` | JSON |
| 文档分块原文 | `.sessions/<name>/chunks.json` | JSON |
| 向量索引 | `faiss_sessions/<hash>/` | FAISS 二进制 |

## 九、课程达标自检

对照 easy-langent 第五章 5.2 和 5.6 节要求：

| 课程要求 | 状态 | 实现方式 |
|---------|------|---------|
| 提示词模板标准化 | ✅ | `prompts.py` 集中管理 |
| 参数化 / 少样本提示 | ✅ | 所有模板参数化 |
| Runnable / `\|` 构建 ≥2 步链 | ✅ | LCEL 链 + 消息列表 Agent 循环 |
| Memory 或 Tool | ✅ 超额 | 分层记忆 + 10 Tool 全做 |
| OutputParser | ✅ | StrOutputParser + JSON + Mermaid |
| 可运行、可说明、可检查 | ✅ | README + 本文档 |
